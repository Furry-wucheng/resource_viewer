## 首页查询管线设计

### 为什么阶段 07 要做底层重构
首个 release 后，用户资源数量会迅速超过开发期样本。如果仍沿用“先取 100 条资源，再在 ViewModel 内过滤/搜索/加载缩略图”的方式，会出现：

- 首页只显示前 100 条，用户以为资源丢失。
- 标签筛选和搜索无法稳定分页。
- 资源越多，启动和筛选越依赖内存过滤，卡顿不可控。
- 缩略图查询范围过大，磁盘访问和 UI rebuild 容易被放大。

因此阶段 07 需要把首页资源查询作为 release 级基础能力一次做完整，而不是只做 UI 层滚动加载。

### 目标架构
保持 MVVM + Repository 约束：

```
HomePage
  -> HomeViewModel
    -> ResourceRepository.queryResources(ResourceQuery)
      -> AppDatabase.queryResources(...)
```

### 新增查询模型

| 模型 | 位置 | 说明 |
|------|------|------|
| `ResourceQuery` | `lib/domain/models/resource_query.dart` 或 `lib/data/repositories/resource_query.dart` | 查询条件：搜索词、标签集合、排序、分页游标、页大小 |
| `ResourceCursor` | 同上 | 键集游标：`createdAt + id`；如支持名称排序则为 `name + id` |
| `PagedResult<T>` | `lib/domain/core/paged_result.dart` | 返回 `items / nextCursor / hasMore / totalCount?` |
| `ResourceSort` | 同上 | `createdDesc`（MVP 必选），`nameAsc`（可选但需预留） |

建议 MVP 首个 release 先强制首页按 `createdAt DESC, id ASC` 排序；名称排序如没有 UI 入口，可只预留枚举和测试骨架。

### SQL 实现思路

查询必须在数据库层完成：

1. **可用源过滤**
   `resources.source_id` join `sources.id`，限制 `sources.enabled = true AND sources.is_available = true AND resources.is_available = true`。

2. **搜索条件**
   `resources.name LIKE ?`，输入为空则不加条件。MVP 可先使用 `%query%`；如后续资源量极大，再升级 FTS。

3. **标签交集筛选**
   当 `tagIds` 非空：
   - join `resource_tags`
   - `resource_tags.tag_id IN (...)`
   - `GROUP BY resources.id`
   - `HAVING COUNT(DISTINCT resource_tags.tag_id) = tagIds.length`

4. **键集分页**
   首屏不带 cursor；下一页使用：
   ```sql
   WHERE (
     resources.created_at < :lastCreatedAt
     OR (resources.created_at = :lastCreatedAt AND resources.id > :lastId)
   )
   ORDER BY resources.created_at DESC, resources.id ASC
   LIMIT :pageSize + 1
   ```
   多取 1 条判断 `hasMore`，返回时截断到 `pageSize`。

5. **计数策略**
   `totalCount` 可单独查询；为减少卡顿，MVP 可在筛选栏显示“当前已加载 N+”，或异步补 `countResources(ResourceQuery without cursor)`。

6. **索引策略**
   现有 `(createdAt, id)` 复合索引可用；需要确认/补充：
   - `resources(source_id, is_available, created_at, id)`
   - `resource_tags(tag_id, resource_id)`
   - `resources(name, id)` 如名称排序或搜索性能需要

### ViewModel 实现思路

`HomeViewModel` 应改为分页状态机：

| 状态字段 | 说明 |
|----------|------|
| `_resources` | 当前已加载资源 |
| `_query` | 当前搜索/标签/排序条件 |
| `_nextCursor` | 下一页游标 |
| `_hasMore` | 是否还有更多 |
| `_isLoadingFirstPage` | 首屏加载 |
| `_isLoadingMore` | 下一页加载 |
| `_loadMoreError` | 下一页错误，不覆盖首屏成功内容 |
| `_queryGeneration` | 搜索/筛选切换时丢弃旧请求结果 |

关键行为：
- `loadFirstPage()`：清空列表、清空缩略图 Map、按当前 query 查询第一页。
- `loadNextPage()`：无更多/正在加载时直接返回。
- `setSearchQuery()`：继续使用 debounce，但触发的是重新加载第一页。
- `toggleTag/selectAll/selectFavorite()`：更新 query 后重新加载第一页。
- 缩略图只加载本次新增 page 的资源，已有资源不重复查。
- 删除/批量删除后，优先从本地列表移除；必要时刷新第一页保证排序和计数正确。

### UI 实现思路

- `ResourceGrid` 接收 `hasMore / isLoadingMore / loadMoreError / onLoadMore`。
- 使用 `ScrollController` 或 `NotificationListener<ScrollNotification>`，距离底部一定阈值触发 `onLoadMore`。
- 底部追加 loading / retry / no more 状态，不影响已加载资源网格。
- 空状态区分：无资源、筛选后无结果、搜索无结果。

### 迁移与风险控制

1. 先补 Repository/Database 查询测试，确认 SQL 语义。
2. 再接 HomeViewModel，保留旧的 `pageResources` 方法直到 UI 稳定。
3. 最后删除或降级旧的内存筛选路径，避免两套逻辑长期并存。
4. 测试覆盖重复 `createdAt`、同名资源、标签交集、删除后分页、搜索切换并发返回。
5. 发布前用脚本造 1k/10k 条资源做手动滚动和筛选冒烟。

## 文件浏览器性能设计

### 当前瓶颈
源文件浏览在 release 前需要重点优化。当前代码已经具备 TTL 缓存和请求去重，但仍有几类卡顿来源：

- `FileBrowserViewModel.loadDirectory` 进入目录后会一次性把目录全部交给 UI；超大目录下 build 和 layout 压力明显。
- `FileListView` / `FileGridView` 的可见项通过 `FutureBuilder` 触发缩略图加载，缺少统一并发限制和取消/优先级控制。
- `_thumbnailFutures` 以 path 持久增长，长时间浏览多个目录会积累大量 Future 和图片字节引用。
- `_loadImportedPaths` 虽已批量加载标签，但仍按数据源加载全部资源；大源下进入任意目录都要建立全源 path 映射。
- `_directoryContainsSupportedContent` 在批量添加时会深度扫描目录，若用户选择很多大目录，容易长时间占用 IO。
- 文件浏览器预览虽然已限制为直属文件，但缓存版本和并发策略仍需作为发布前体验项固定下来。

### 优化目标

| 方向 | 目标 |
|------|------|
| 目录加载 | 大目录进入后尽快显示首屏，避免一次性 UI 压力 |
| 缩略图 | 只加载可见区域附近缩略图，并限制并发 |
| 入库状态 | 只为当前目录可见路径建立状态，避免每次加载全源映射 |
| 深扫任务 | 批量添加前的目录内容检测可取消、可限并发、可展示进度 |
| 缓存生命周期 | 目录切换时释放无关缩略图 Future 和内存字节 |
| 远程源 | SMB 等远程源减少重复 listDirectory，避免缩略图任务压垮连接 |

### 实现思路

1. **目录条目分页/分批渲染**
   - `FilesystemRepository.listDirectory` 仍返回完整目录列表，先不强行要求所有 FileSource 支持分页。
   - `FileBrowserViewModel` 增加展示窗口：`_allEntries` + `_visibleEntries` + `loadMoreEntries()`。
   - 首次只展示 100~200 条，滚动到底部再追加。
   - 列表/网格 UI 的多选、全选默认作用于“当前目录全部条目”还是“当前已展示条目”需明确；建议 release 中全选当前已展示条目，并在 UI 文案中体现。

2. **缩略图加载队列**
   - 新增 `ThumbnailPreviewQueue` 或在 `FileBrowserViewModel` 内封装队列。
   - 限制并发：本地源 4~6，远程源 1~2。
   - 任务按可见区域优先；滚远后旧任务结果可丢弃。
   - 缩略图失败记录短 TTL，避免同一路径反复失败重试。
   - 目录切换时清理 `_thumbnailFutures`，只保留当前目录和可选的上一层缓存。

3. **可见项缩略图触发**
   - `FileGridView` / `FileListView` 不直接无限制调用 `thumbnailLoader(entry)`。
   - 增加 `cacheExtent` 或滚动监听，只对可见项和近邻项请求缩略图。
   - 列表模式可默认不生成文件夹预览，或只在网格模式生成，减少列表滚动 IO。

4. **入库状态按当前目录查询**
   - 新增 Repository 方法：`getResourcesBySourceIdAndPaths(sourceId, paths)` 或 `getImportedResourceSummaries(sourceId, paths)`。
   - `loadDirectory` 只为当前目录条目路径查询入库状态和标签。
   - 批量添加成功后局部更新当前目录状态，不重新扫全源。
   - 如果需要“父目录下子资源已入库”的提示，作为后续增强，不阻塞 release。

5. **批量添加深扫优化**
   - `_directoryContainsSupportedContent` 改为限并发 BFS/DFS，并提供进度。
   - 对用户选择的多个目录串行或小并发处理，避免同时深扫多个大树。
   - 增加最大扫描节点保护；超过阈值时提示用户“目录较大，是否继续深度检查”。
   - 深扫只发生在用户确认批量添加后，不在普通浏览时触发。

6. **缓存失效与刷新**
   - 保留 `FilesystemRepository` TTL 缓存和请求去重。
   - 增加当前目录手动刷新按钮或下拉/菜单刷新，调用 `invalidateCache(sourceId)` 后重载当前路径。
   - 源配置变更、批量添加完成、删除资源后局部刷新相关状态。

### 验收数据

发布前至少做三类目录冒烟：

| 场景 | 验收 |
|------|------|
| 1k 文件本地目录 | 进入目录首屏可快速显示，滚动追加不卡死 |
| 10k 文件本地目录 | 不一次性渲染全部卡死；滚动和返回可操作 |
| SMB/慢 IO 模拟 | 缩略图逐步出现，UI 可继续滚动和返回 |

## 7.8 文件浏览器性能重构

### 执行目标
系统优化源文件浏览，确保大目录、远程源、缩略图加载和批量添加深扫不会阻塞 UI。

### 任务边界

- 普通浏览不得深度扫描目录内容。
- 批量添加可以深扫，但必须限并发、可反馈进度。

### 产出物
| 文件 | 说明 |

| `filesystem_repository.dart` | 如有需要补按路径查询/刷新能力 |
| `resource_repository.dart` | 当前目录路径批量查询入库资源 |

| `batch_add_resources_dialog.dart` | 批量添加进度/大目录提示（如放在 Dialog 内） |

### 验收标准
- [ ] 进入 1k 文件目录首屏快速显示，不等待全部缩略图完成
